# 実装・デプロイ手順書 — 篠田の妖精さん LINE bot

- 版: 1.0 (2026-06-11)
- 対象: 実装担当エージェント/開発者
- 構成: LINE Messaging API(無料コミュニケーションプラン) + Cloudflare Workers(無料枠) + Workers KV

---

## Phase 1: LINE側の準備

### 1-1. LINE公式アカウント & チャネル作成

1. [LINE Developersコンソール](https://developers.line.biz/console/) にLINEアカウントでログイン
2. プロバイダーを新規作成(名前は任意。例: `shinoda`)
3. 「Messaging APIチャネル」を作成
   - チャネル名: `篠田の妖精さん`(任意。LINE上のbot表示名になる)
   - 業種等は適当でよい
   - ※ 2024年以降はLINE公式アカウント作成→Messaging API有効化のフローに誘導される場合がある。その場合は [LINE Official Account Manager](https://manager.line.biz/) でアカウント作成後、設定 > Messaging API から有効化してチャネルを紐付ける
4. 料金プランは**コミュニケーションプラン(0円/月200通)** のままにする

### 1-2. ボット設定(LINE Official Account Manager > 設定 > 応答設定)

| 設定 | 値 | 理由 |
|---|---|---|
| 応答メッセージ(自動応答) | **オフ** | botの返答と二重になるため |
| あいさつメッセージ | 任意(オフ推奨) | JOIN_GREETINGで代替 |
| Webhook | **オン** | 必須 |
| **トークルームへの参加(グループ)** | **許可** | グループLINEで使うため必須 |

### 1-3. 認証情報の取得(LINE Developersコンソール)

- **チャネルシークレット**: チャネル基本設定タブ → `LINE_CHANNEL_SECRET`
- **チャネルアクセストークン(長期)**: Messaging API設定タブで発行 → `LINE_CHANNEL_ACCESS_TOKEN`

Webhook URLはWorkerデプロイ後(Phase 3)に設定する。

## Phase 2: Cloudflare側の準備

### 2-1. アカウントとツール

```bash
# Cloudflareアカウント作成(無料): https://dash.cloudflare.com/sign-up
npm install -g wrangler   # または npx wrangler
wrangler login
```

### 2-2. プロジェクト作成

```bash
npm create cloudflare@latest shinoda-fairy-bot -- --type hello-world --ts
cd shinoda-fairy-bot
```

### 2-3. KV namespace 作成

```bash
wrangler kv namespace create STATE
# 出力された id を wrangler.toml に転記
```

### 2-4. wrangler.toml(雛形)

```toml
name = "shinoda-fairy-bot"
main = "src/index.ts"
compatibility_date = "2026-06-01"

[triggers]
crons = ["*/10 * * * *"]   # UTC基準・10分毎

kv_namespaces = [
  { binding = "STATE", id = "<2-3で取得したID>" }
]

[vars]
GOMI_DAYS = "2,5"            # 火,金 (0=日)
GOMI_START_HOUR = "20"       # JST
GOMI_END_HOUR = "23"         # JST
LAUNDRY_DURATION_MIN = "50"
LAUNDRY_NAG_INTERVAL_MIN = "30"
LAUNDRY_GIVEUP_AFTER_MIN = "180"
SHOPPING_HOURS = "15,18,21"  # JST
INVALID_STREAK_THRESHOLD = "3"
INVALID_RANDOM_RATE = "0.05"
```

### 2-5. シークレット登録

```bash
wrangler secret put LINE_CHANNEL_SECRET
wrangler secret put LINE_CHANNEL_ACCESS_TOKEN
```

## Phase 3: 実装 → デプロイ

### 3-1. 実装

`02_設計書.md` の §4(コマンド判定)・§5(Cron)・§6(状態遷移)・§10(プロジェクト構成)に従って実装する。
`03_メッセージ定型文集.md` §2 を `src/messages.ts` に変換して組み込む。

実装チェックリスト(設計書の受け入れ基準に対応):

- [ ] `POST /webhook` で署名検証(`crypto.subtle` の HMAC-SHA256 + Base64 比較)
- [ ] `mention.mentionees[].isSelf` でメンション判定、メンション文字列の除去
- [ ] §4 の優先順位どおりのコマンドルーター
- [ ] reply API での応答(入力への返答はすべてreply)
- [ ] `scheduled` ハンドラでJST換算 → ゴミ/洗濯/買い出しの期限判定 → push
- [ ] 状態変更時のみ KV `put`
- [ ] join/leave イベント処理(groups登録/削除)
- [ ] 買い出しのクイックリプライ(postback: `shopping_day?id=...&day=today|tomorrow`)

### 3-2. ローカル動作確認

```bash
wrangler dev
# 別ターミナルから webhook を模擬(署名検証を一時スキップするデバッグフラグを用意すると楽)
# Cronの手動発火:
wrangler dev --test-scheduled
curl "http://localhost:8787/__scheduled?cron=*/10+*+*+*+*"
```

ユニットテスト(任意だが推奨): `vitest` + `@cloudflare/vitest-pool-workers` でコマンド判定と状態遷移をテスト。

### 3-3. デプロイ

```bash
wrangler deploy
# => https://shinoda-fairy-bot.<account>.workers.dev が発行される
```

### 3-4. Webhook URL の設定

1. LINE Developersコンソール > Messaging API設定タブ
2. Webhook URL: `https://shinoda-fairy-bot.<account>.workers.dev/webhook`
3. 「検証」ボタンで疎通確認(Workerは `GET`/検証リクエストに `200` を返すこと)
4. 「Webhookの利用」をオン

## Phase 4: グループへの導入と受け入れテスト

1. botを友だち追加(Messaging API設定タブのQRコード)
2. 篠田のグループLINEにbotを招待 → JOIN_GREETING + help が届くこと
3. 受け入れテスト(`01_要件定義書.md` §6 のチェックリストを実施):

| # | 操作 | 期待結果 |
|---|---|---|
| 1 | メンションなしで「洗濯」と発言 | 反応しない |
| 2 | @妖精 ヘルプ | コマンド一覧が返る |
| 3 | @妖精 洗濯 | 終了予定時刻(+50分)が返る |
| 4 | 40分後/50分後 | 「あと10分」「完了」が届く(±10分) |
| 5 | 放置30分 | 「早く干そうね」系が届く |
| 6 | @妖精 干した | 褒められ、催促が止まる |
| 7 | @妖精 牛乳を買ってくる | 今日/明日のクイックリプライが出る |
| 8 | 「今日」を選択(15時以降) | 即「牛乳を買ってきてね」が返る |
| 9 | @妖精 牛乳買った | 褒められ、催促が止まる |
| 10 | 品目を3件登録後に4件目 | 上限の旨が返る |
| 11 | @妖精 あああ ×3回 | 1-2回目は呆れ、3回目はキレ反応 |
| 12 | 火or金 20時台 | 「今日はゴミ捨ての日だよ」が届く |
| 13 | @妖精 まとめた → 21時以降 | 「まだ捨ててないのかな？」系が届く |
| 14 | @妖精 捨てた | 褒められ、当日のpushが止まる |

時刻依存のテストは `GOMI_DAYS` / `GOMI_START_HOUR` / `SHOPPING_HOURS` を一時的に現在時刻付近へ変更(`wrangler.toml` 編集 → `wrangler deploy`)すると短時間で検証できる。

## Phase 5: 運用

- **メッセージ残量の確認**: LINE Official Account Manager のダッシュボード、または `GET https://api.line.me/v2/bot/message/quota/consumption`
- 200通/月に近づく場合の調整(設計書 §9): `GOMI_END_HOUR` を下げる、`LAUNDRY_GIVEUP_AFTER_MIN` を減らす、`SHOPPING_HOURS` を `"15,21"` にする等
- Cloudflare側の利用量: ダッシュボード > Workers。本用途では無料枠(10万req/日、KV書込1,000/日)に達することはまずない
- 文言追加: `03_メッセージ定型文集.md` を更新 → `messages.ts` 再生成 → `wrangler deploy`

## トラブルシューティング

| 症状 | 原因候補 |
|---|---|
| webhook検証が失敗する | URLパス間違い / Workerが200を返していない / デプロイ未完了 |
| メンションしても無反応 | 署名検証失敗(secret誤り) / 応答設定でWebhookがオフ / isSelf判定漏れ |
| リマインドが来ない | Cron未設定 / JST換算ミス(UTCのまま判定) / `groups` にgroupId未登録(joinイベント前に招待済みの場合は一度退会→再招待) |
| 月途中でpushが届かなくなった | 200通超過(429/403エラー)。翌月1日に回復 |
| 通知が最大10分遅れる | 仕様(Cron間隔)。問題なら間隔を `*/5` に短縮可(無料枠内) |
