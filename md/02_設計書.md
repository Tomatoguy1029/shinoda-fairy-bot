## 1. アーキテクチャ

```
LINEグループ
   │  ① メンション付きメッセージ / postback
   ▼
LINE Platform ──(Webhook POST)──▶ Cloudflare Worker
                                     ├─ fetch handler   : 署名検証 → コマンド判定 → reply送信 + KV更新
                                     └─ scheduled handler: Cron(10分毎, UTC) → JST換算 → 期限到来ジョブ判定 → push送信 + KV更新
                                     ▼
                               Workers KV (状態保存)
```

- ランタイム: Cloudflare Workers(無料枠)。フレームワークなしの素の `fetch`/`scheduled` ハンドラで十分(Hono利用は任意)
- 言語: TypeScript
- LINE SDKは使わず、Messaging API REST を直接呼ぶ(依存最小化。`@line/bot-sdk` 利用も可)
- Cronトリガー: `*/10 * * * *`(10分毎・UTC)。1個のみ使用

### 1.1 環境変数 / バインディング

| 名前 | 種別 | 内容 |
|---|---|---|
| `LINE_CHANNEL_SECRET` | secret | 署名検証用 |
| `LINE_CHANNEL_ACCESS_TOKEN` | secret | API呼び出し用(長期トークン) |
| `STATE` | KV namespace | 状態保存 |
| `GOMI_DAYS` | var | ゴミの日(0=日..6=土)。デフォルト `"2,5"`(火・金) |
| `GOMI_START_HOUR` / `GOMI_END_HOUR` | var | デフォルト `20` / `23` (JST) |
| `LAUNDRY_DURATION_MIN` | var | デフォルト `50` |
| `LAUNDRY_NAG_INTERVAL_MIN` | var | デフォルト `30` |
| `LAUNDRY_GIVEUP_AFTER_MIN` | var | デフォルト `180`(完了後この分数で催促を自動停止) |
| `SHOPPING_HOURS` | var | リマインド時刻。デフォルト `"15,18,21"` (JST) |
| `INVALID_STREAK_THRESHOLD` | var | キレ反応の連続閾値。デフォルト `3` |
| `INVALID_RANDOM_RATE` | var | キレ反応の確率。デフォルト `0.05` |

## 2. LINE API 利用方針

| 用途 | エンドポイント | 課金 |
|---|---|---|
| ユーザー入力への返答(すべて) | `POST /v2/bot/message/reply` | **無料・無制限** |
| リマインド | `POST /v2/bot/message/push`(宛先=groupId) | 月200通枠を消費(**グループ人数分**) |
| 残量確認(任意) | `GET /v2/bot/message/quota/consumption` | 無料 |

**原則: replyで返せるものは必ずreplyで返す。pushはCron起点のリマインドのみ。**

### 2.1 Webhook署名検証

`x-line-signature` ヘッダと、`HMAC-SHA256(channelSecret, requestBody)` の Base64 を比較。不一致は `403`。Workers では `crypto.subtle` を使用。検証成功時は即 `200` を返しつつ処理する(LINEはタイムアウトに厳しいため、`event.waitUntil` 利用可。ただしreply処理は軽量なので同期でも実用上問題ない)。

### 2.2 メンション検知

`message` イベント(type=text)で、`event.message.mention?.mentionees?.some(m => m.isSelf === true)` の場合のみコマンド処理する。
コマンド解析前に、メンション部分(`mentionees` の `index`/`length` で示される範囲)をテキストから除去し、前後空白をtrimする。

- メンションなしテキスト → 無視(200のみ)
- `postback` イベント → メンション概念がないため常に処理(買い出しの日選択に使用)
- `join` イベント(グループ招待時) → groupId を登録し、挨拶+help を reply

## 3. KV データモデル

書込1,000/日の制約に対応するため、**1グループ=1キー**にすべての状態を集約し、変更があった時のみ `put` する。

- キー: `g:{groupId}`
- 登録済みグループ一覧キー: `groups`(JSON配列。joinイベントで追加)

```ts
interface GroupState {
  groupId: string;
  gomi: {
    date: string;            // "YYYY-MM-DD" (JST) この状態が属するゴミの日
    status: "pending" | "bundled" | "done";
    announced: boolean;      // 初回アナウンス送信済みか
    lastPushHour: number | null; // 当日最後にpushしたJST時(重複防止)
  };
  laundry: null | {
    startedAt: number;       // epoch ms
    almostNotified: boolean; // +40分通知済み
    doneNotified: boolean;   // +50分通知済み
    lastNagAt: number | null;
  };
  shopping: ShoppingItem[];  // 最大3件
  invalidStreak: number;
}

interface ShoppingItem {
  id: string;                // crypto.randomUUID()
  name: string;              // 品目名
  status: "awaiting_day" | "active";
  startDate: string | null;  // リマインド開始日 "YYYY-MM-DD" (JST)
  lastPushHour: number | null; // 当日最後にpushしたJST時
  rolledOver: boolean;       // 前日から繰り越された品目か
  createdAt: number;
}
```

注意: `awaiting_day` の品目が残ったまま新規登録が来た場合は、古い `awaiting_day` を破棄して新しい質問を優先する(同時に複数の日付質問を抱えない)。

## 4. コマンド判定仕様(fetchハンドラ)

メンション除去後のテキスト `t` に対し、**以下の優先順で最初にマッチしたものを実行**する。正規化: 全角英数→半角、大文字→小文字、空白除去はせず trim のみ。

| 優先 | コマンド | 判定(正規表現/条件) | アクション |
|---|---|---|---|
| 1 | ヘルプ | `/^(help|ヘルプ|へるぷ)$/i` | `HELP` をreply |
| 2 | 日選択(テキスト) | `shopping` に `awaiting_day` があり、`/今日|きょう/` または `/明日|あした|あす/` | §6.3 の日確定処理 |
| 3 | 洗濯・干した | `/(干した|ほした|乾かした|干したよ)/` | 洗濯タイマー停止→`LAUNDRY_HUNG_PRAISE` をreply。タイマーなしなら `LAUNDRY_NOT_RUNNING` |
| 4 | 洗濯開始 | `/洗濯|せんたく/` (かつ「干」を含まない) | §6.2 タイマー開始→`LAUNDRY_START`(終了予定時刻埋め込み)をreply。既に稼働中なら `LAUNDRY_ALREADY` |
| 5 | ゴミ捨て完了 | `/(捨てた|すてた|出した|だした)/` | §6.1 status→done。`GOMI_DONE_PRAISE` をreply。ゴミの日以外/既にdoneなら `GOMI_NOT_TODAY` / `GOMI_ALREADY_DONE` |
| 6 | ゴミまとめ完了 | `/(まとめた|まとめました|縛った|しばった)/` | status→bundled。`GOMI_BUNDLED_ACK` をreply(同上の例外反応) |
| 7 | 買い出し完了(品目指定) | `/^(.+?)(を)?(買った|かった|買ってきた)$/` | 品目名で `active`/`awaiting_day` を前方一致・部分一致検索→完了。見つからなければ `SHOPPING_ITEM_NOT_FOUND` |
| 8 | 買い出し完了(無指定) | `/^(買った|かった|買ってきた)$/` | 登録1件→完了。複数件→`SHOPPING_WHICH_ONE`(どれ？と聞き返す)。0件→`SHOPPING_NOTHING` |
| 9 | 買い出し登録 | `/^(.+?)(を)?(買ってくる|買いに行く|買う|買っとく|買わなきゃ)$/` | §6.3 登録処理 |
| 10 | 品目名のみ(完了) | `t` が登録済み品目名と完全一致 or 品目名を含む | 該当品目を完了 |
| 11 | 無効入力 | 上記いずれにも該当しない | §6.4 |

- 有効コマンド(1〜10)処理時は `invalidStreak = 0` にリセット
- 返答文はすべて該当シチュエーションのプールからランダム選択(§7)

## 5. Cron処理仕様(scheduledハンドラ)

10分毎に起動。`now` をJSTに換算し(`jst = new Date(Date.now() + 9*3600*1000)` をUTCメソッドで読む)、`groups` の各グループについて以下を順に評価。**状態に変化があった場合のみKVへput**。pushは1グループの1回のCron実行で複数件発生しうる(まとめて1リクエスト・最大5メッセージオブジェクトに集約してもよいが、通数は宛先人数×リクエスト回数ではなく宛先人数×送信回数なので、**1回のpushに複数バブルをまとめると通数節約になる**。実装は「同一Cron実行内のpushは1リクエストに集約」とする)。

### 5.1 ゴミ出し

```
todayJst = "YYYY-MM-DD"; dow = JSTの曜日
if (dowがGOMI_DAYSに含まれない) → 何もしない
if (gomi.date != todayJst) → gomiをリセット {date: todayJst, status: "pending", announced: false, lastPushHour: null}
if (status == "done") → 何もしない
h = JSTの時
if (h < GOMI_START_HOUR || h > GOMI_END_HOUR) → 何もしない
if (gomi.lastPushHour == h) → 何もしない   // 1時間に1回だけ
push対象:
  !announced            → GOMI_ANNOUNCE      (announced=true)
  status == "pending"   → GOMI_NAG_PRE_BUNDLE
  status == "bundled"   → GOMI_NAG_POST_BUNDLE
※ 初回(20時台)は GOMI_ANNOUNCE のみ送る(まだ呆れない)。21時以降にステータス別の催促。
gomi.lastPushHour = h
```

### 5.2 洗濯

```
if (laundry == null) → 何もしない
elapsed = now - laundry.startedAt
if (!almostNotified && elapsed >= 40分) → push LAUNDRY_ALMOST; almostNotified = true
if (!doneNotified && elapsed >= 50分)   → push LAUNDRY_DONE;   doneNotified = true; lastNagAt = now
if (doneNotified):
    if (now - 完了時刻 >= LAUNDRY_GIVEUP_AFTER_MIN分)
        → push LAUNDRY_GIVEUP; laundry = null   // 完了後3時間で自動停止
    else if (now - lastNagAt >= 30分)
        → push LAUNDRY_NAG; lastNagAt = now
```

注: 10分毎Cronのため通知は最大±10分ずれる。+40分通知と+50分通知が同一実行で両方期限到来した場合は**完了通知のみ**送る(almost をスキップして両フラグを立てる)。

### 5.3 買い出し

```
// 繰り越し: 指定日の25時(=翌1時)を過ぎても未完了なら翌日へ
if (JSTの時 >= 1):
  for item in shopping where status == "active" && item.startDate < todayJst:
      item.startDate = todayJst; item.rolledOver = true
      // 翌15時の再開初回は SHOPPING_ROLLOVER を使用し、以降は通常のNAGに戻す

for item in shopping where status == "active":
  if (todayJst < item.startDate) → skip
  h = JSTの時
  if (h が SHOPPING_HOURS のいずれか && item.lastPushHour != h):
      初回(lastPushHour == null) → push SHOPPING_FIRST_REMIND(item.name)
      2回目以降                  → push SHOPPING_NAG(item.name)
      item.lastPushHour = h
※ 日付が変わったら lastPushHour は意味を失うため、「lastPushHour != h」判定に加えて
  「最終push日 != today なら初回扱いにしない(NAG継続)」とする。実装は lastPushHour を
  "YYYY-MM-DD-HH" 文字列にすると簡潔。
※ 複数品目が同時刻に該当する場合は1つのpushリクエストにまとめる(例:「牛乳と卵、買ってきてね」
  …実装簡略化のため、品目を「、」連結して1テンプレートに埋め込む方式でよい)。
```

「今日買う」が選択され、かつ選択時点のJSTが15時以降の場合は、**Cronを待たず選択処理の中で即座に初回リマインドをreplyで返す**(reply可能なため無料)。その際 `lastPushHour` に現在時を記録し、直近のCron重複pushを防ぐ。

## 6. 各機能の状態遷移詳細

### 6.1 ゴミ出し

```
pending ──「まとめた」──▶ bundled ──「捨てた」──▶ done
pending ──「捨てた」(まとめ報告を飛ばす)──▶ done   ※直接doneも許可
ゴミの日以外の「まとめた/捨てた」 → 状態変更せず GOMI_NOT_TODAY をreply
```

### 6.2 洗濯

```
(なし) ──「洗濯」──▶ running(startedAt記録)
running ──「洗濯」──▶ LAUNDRY_ALREADY をreply(タイマー再セットはしない)
running/完了後 ──「干した」──▶ (なし) + 褒め
完了後催促が LAUNDRY_NAG_MAX 回 ──▶ (なし) 自動打ち切り
```

終了予定時刻 = `startedAt + LAUNDRY_DURATION_MIN分` をJSTの `HH:MM` で `LAUNDRY_START` テンプレートに埋め込む。

### 6.3 買い出し

```
「〇〇を買ってくる」
  ├ 登録済み3件 → SHOPPING_FULL をreply(登録しない)
  ├ 同名品目が既に登録済み → SHOPPING_DUPLICATE をreply
  └ 登録 {status: awaiting_day} + SHOPPING_ASK_DAY をreply
       └ クイックリプライ: [今日] [明日] (postback action)
            data: "shopping_day?id={itemId}&day=today|tomorrow"
「今日」選択 → startDate = 今日, status = active
              + 現在15時以降なら即 SHOPPING_FIRST_REMIND をreply(§5.3)
              + 15時前なら SHOPPING_REGISTERED_TODAY をreply
「明日」選択 → startDate = 翌日, status = active, SHOPPING_REGISTERED_TOMORROW をreply
完了入力 → 品目を配列から削除 + SHOPPING_DONE_PRAISE(品目名埋め込み)をreply
           全品目が空になったら追加で一言(SHOPPING_ALL_CLEAR を連結)してもよい
```

postbackの `id` が既に存在しない(破棄済み/完了済み)場合は `SHOPPING_STALE_CHOICE` をreply。

### 6.4 無効入力

```
invalidStreak++
if (invalidStreak >= INVALID_STREAK_THRESHOLD || random() < INVALID_RANDOM_RATE)
    → INVALID_ANGRY プールからreply
else
    → INVALID_BASIC プールからreply
```

`INVALID_ANGRY` には「は？」「何を言ってるの？」「ポニョ」および壊れた機械風の文字列(定型文集に収録)を含む。

## 7. メッセージ選択ロジック

- `messages.ts` に `Record<SituationId, string[]>` としてプールを保持(`03_メッセージ定型文集.md` §2 のID体系に従う)
- 選択: 一様ランダム。**直前に使った文言の連続回避**は必須としない(任意で実装可)
- プレースホルダ: `{item}`(品目名)、`{time}`(HH:MM)、`{items}`(品目列挙)を `replace` で埋める

## 8. エラーハンドリング / エッジケース

| ケース | 対応 |
|---|---|
| push送信が429/エラー(月200通超過) | ログのみ。リトライしない(翌月回復)。状態更新は行い、無限再送を防ぐ |
| reply tokenの期限切れ・二重使用 | ログのみ |
| KV書込失敗 | そのCron回はスキップ(次回再評価される設計のため自己回復する) |
| グループから退会(leaveイベント) | `groups` から groupId を削除し、`g:{groupId}` を削除 |
| 複数グループ参加 | 設計上対応済み(groups配列を走査)。ただし無料200通枠は全グループ合算な点に注意 |
| テキスト以外(スタンプ等)でメンション | メンション情報が付かないため無視される(問題なし) |

## 9. push通数の試算(無料200通枠)

前提: グループ人数 N(篠田ビル住人数)。**1回のpush = N通消費**。

| 機能 | 最悪ケース/月 | push回数 |
|---|---|---|
| ゴミ出し | 週2日×4.5週×最大4回(20,21,22,23時) | 36回 |
| 洗濯 | 週3回洗濯×(2+6催促) | 24回 |
| 買い出し | 月10品目×3回/日×2日 | 60回 |

合計最悪 ~120回/月 → **N=1なら120通でセーフ、N=2で240通となり超過リスク**。

**対策(設計に組込済み・運用で調整)**
1. 「捨てた」「干した」「買った」を早く報告すれば催促は止まる(本来の利用)
2. `GOMI_END_HOUR`、`LAUNDRY_GIVEUP_AFTER_MIN`、`SHOPPING_HOURS` で上限を絞れる
3. 同一Cron実行内のpushは1リクエストに集約(§5)
4. 任意実装: Cron内で `GET /quota/consumption` を確認し、残量が閾値(例: 残20通)を切ったら催促系pushを停止、初回アナウンスのみ継続
5. 人数が多い場合の代替: botを1対1トークでも友だち登録し、リマインドは代表者1人へのpushにする運用も可(将来検討)

## 11. テスト観点

- 署名検証: 正しい/誤った署名でのレスポンスコード
- コマンド判定: §4 の各正規表現に対する代表入力・境界入力(「洗濯物干した」→干した判定が優先、など)のユニットテスト
- 状態遷移: §6 の遷移表を網羅
- Cron: 時刻をモックして §5 の発火条件(JST境界、日付跨ぎ、重複防止)を検証
- `wrangler dev --test-scheduled` で `curl "http://localhost:8787/__scheduled?cron=*/10+*+*+*+*"` によりCronを手動発火可能
